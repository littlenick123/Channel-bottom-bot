# 草稿确认、批量按钮、槽位命名与频道自动发现设计

日期：2026-07-20

## 目标

在现有纯 Bot API、aiogram、SQLite 单进程架构上补充四项管理能力：转发内容保存前确认、URL 按钮批量添加、频道槽位可辨识名称，以及机器人成为频道管理员时自动记录频道 ID 并绑定操作人。

本次修改保持现有多租户权限隔离、不可变草稿版本、频道共享槽位快照、按大编号到小编号发布和单频道防抖刷新机制不变。

## 用户交互

### 转发内容确认

用户在私聊中转发单条帖子或相册后，机器人不立即创建正式草稿，而是：

1. 将内容无来源复制到私密存储频道。
2. 建立有效期为 10 分钟的待确认草稿记录。
3. 向用户发送无转发来源的内容预览。
4. 显示“保存为草稿”“保存并命名”“放弃”三个回调按钮。

“保存为草稿”使用正文摘要或“媒体草稿”作为默认名称。“保存并命名”进入名称输入状态；名称最多 100 个字符。“放弃”立即删除待确认记录并尝试清理对应的存储频道消息。

在“保存并命名”状态使用 `/cancel` 时，只取消名称输入，不放弃待确认内容；机器人重新显示三个确认操作。用户必须明确点击“放弃”或等待过期才会删除临时内容。

相册按现有 0.8 秒聚合窗口作为一个整体进入确认流程。待确认内容不占用正式草稿配额；确认时才检查每用户草稿上限。达到上限时保留待确认内容直到过期，用户清理旧草稿后可以再次确认。

待确认记录持久化到 SQLite，因此机器人重启后原确认按钮仍可使用。确认操作必须幂等：同一个待确认记录只能转为一个正式草稿，重复点击返回“已经处理或已过期”。

### URL 按钮批量添加

草稿按钮编辑器接受一次多行输入，每个非空行使用：

```text
按钮文字 | URL | 行号
```

例如：

```text
官网 | https://example.com | 1
客服 | tg://resolve?domain=example | 1
下载 | https://example.com/download | 2
```

批量添加采用追加语义。每个按钮的列号根据目标行已有按钮数量和本批前序按钮自动计算。整批在写入前统一验证：格式完整、行号为正整数、URL 协议为 `https://`、`http://` 或 `tg://`、每行最多 8 个、总计最多 100 个。任何一项失败都不创建草稿新版本；全部通过后只创建一个新版本。

现有“清空全部按钮”操作保留。

### 频道槽位名称

每个已占用频道槽位增加一个频道共享的 `display_name`：

- 首次把草稿分配到槽位时，复制当时的草稿名称。
- 未经手动改名的槽位在替换草稿时同步为新草稿名称。
- 管理员手动改名后设置 `name_customized=1`，以后替换内容仍保留该名称。
- 移动或交换槽位时，名称、是否自定义、启停状态和草稿版本快照作为同一个槽位配置整体移动。
- 清空槽位时一并删除名称和自定义状态。

频道页面使用 `1. 活动入口｜已启用｜版本 2` 的形式展示。槽位选择按钮包含编号和截断后的名称，并提供“修改槽位名称”操作。槽位名称最长 100 个字符，去除首尾空白后不得为空。

槽位名称属于频道共享配置，不依赖个人草稿后续是否重命名或删除，也不会改变不可变草稿版本和发布内容。

### 频道自动记录与绑定

应用注册 aiogram 的 `my_chat_member` 更新处理器。机器人在频道中被加入或升为管理员时：

1. 从更新中取得频道 ID、名称、用户名和执行操作的用户。
2. 如果频道 ID 等于 `STORAGE_CHANNEL_ID`，直接忽略目标频道绑定。
3. 写入或更新频道基本资料，以记录真实的 `-100...` ID。
4. 验证执行操作的用户当前是频道创建者或管理员。
5. 验证机器人当前是管理员并拥有发帖和删除消息权限。
6. 在用户频道配额允许时，幂等地绑定执行操作的用户，并写入 `channel.auto_bind` 审计记录。

如果频道已经存在共享配置，只增加经过验证的管理员关联，不创建重复频道。执行操作的用户即使从未私聊机器人，也可以由成员更新中的用户资料建立用户记录；以后发送 `/start` 时会在“我的频道”看到该频道。

机器人只能向曾经主动联系过它的用户发送私聊通知。通知失败不会回滚已经完成的自动绑定。成功通知内容包含频道名称和频道 ID。

机器人被降权、移除或失去发帖/删除权限时，保留已有频道、槽位和管理员关联，把频道状态设为暂停并记录原因。该事件不会创建一个此前不存在的目标频道配置。已绑定管理员在恢复权限后仍可使用现有“检查权限并恢复”操作。

## 组件边界

### `PendingDraftService`

负责准备待确认内容、确认默认命名、确认自定义命名、放弃和清理过期记录。它依赖 Repository 和存储网关，不负责构造 Telegram 菜单。

固定接口：

```python
async def prepare(user_id: int, messages: Sequence[IncomingContent]) -> PendingDraft
async def confirm(pending_id: int, user_id: int, name: str | None = None) -> Draft
async def discard(pending_id: int, user_id: int) -> bool
async def cleanup_expired(now: float, limit: int = 100) -> int
```

正式 `DraftService.capture` 保留供显式直接创建场景和既有调用者使用，但转发处理器不再调用它。待确认转换为正式草稿的事务由 Repository 提供，`DraftService` 继续负责正式草稿的版本修改、复制和垃圾回收。

存储网关新增从存储频道向指定私聊复制待确认内容的预览方法。它使用存储消息 ID，因此预览不带转发来源，并保持相册关系和 Telegram 格式实体。

### 批量按钮解析器

把按钮文本解析从单项 `parse_button_input` 提升为纯函数 `parse_button_batch(value, existing_buttons)`，返回已经分配行列位置的 `tuple[ButtonSpec, ...]`。领域对象 `DraftRevision` 继续作为每行 8 个和总数 100 个的最终防线。

### 槽位命名

`ChannelService.assign_slot` 在分配时取得草稿名称并决定是否更新槽位名称。Repository 负责在同一写操作中保存版本快照引用和名称状态。新增 `ChannelService.rename_slot` 执行权限验证和审计。

### `ChatMembershipService`

接收 `ChatMemberUpdated`，识别管理员状态变化，调用现有权限服务和频道仓库完成自动绑定或暂停。它捕获配额、权限和 Telegram API 错误并返回可记录的结果，不让单个成员更新中断 polling。

## 数据库迁移

新增一个版本化迁移，包含：

```sql
CREATE TABLE pending_drafts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    status TEXT NOT NULL DEFAULT 'pending',
    expires_at REAL NOT NULL,
    created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE pending_draft_items (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    pending_draft_id INTEGER NOT NULL REFERENCES pending_drafts(id) ON DELETE CASCADE,
    position INTEGER NOT NULL,
    text TEXT,
    storage_message_id INTEGER,
    media_kind TEXT,
    telegram_file_id TEXT,
    grouped_id TEXT,
    formatting_entities_json TEXT NOT NULL DEFAULT '[]',
    UNIQUE (pending_draft_id, position)
);

ALTER TABLE channel_slots ADD COLUMN display_name TEXT NOT NULL DEFAULT '';
ALTER TABLE channel_slots ADD COLUMN name_customized INTEGER NOT NULL DEFAULT 0;
```

迁移后既有槽位名称通过对应草稿名称回填，无法取得名称时使用“置底帖子 N”。迁移必须和现有版本记录一起事务化，失败时不提升 schema 版本。

待确认草稿确认时，在一个 `BEGIN IMMEDIATE` 事务中检查用户归属、状态、有效期和配额，创建正式草稿及第一个版本，复用其存储消息 ID，然后删除待确认记录。存储消息只在正式草稿和槽位快照都不再引用时进入既有垃圾回收。

## 生命周期与清理

应用启动后立即执行一次过期待确认清理，再启动每 60 秒执行一次的维护任务。关闭时取消并等待维护任务，方式与刷新调度任务一致。

放弃或过期清理分两步进行：先在数据库中把 `status` 改为 `discarded` 或 `expired` 并取得存储消息 ID，再请求 Telegram 删除。删除失败时保留该终态记录作为可重试清理标记，不能恢复为 `pending`。删除成功或 Telegram 返回消息不存在后才删除数据库记录。

发送用户预览或确认菜单失败时，待确认记录仍保留；用户后续再次转发不会错误确认旧内容。每个确认菜单的回调数据只包含短操作码和整数待确认 ID。

## 错误处理与安全

- 所有待确认、草稿和槽位操作都验证当前用户所有权或频道管理权限。
- 自动绑定只关联 `my_chat_member.from_user`，不会自动授权频道内所有管理员。
- 超级群组成员更新不触发自动绑定；当前自动刷新监听仍以频道帖子为范围。
- 自动绑定遇到用户频道配额时记录频道 ID和失败原因，但不绕过配额创建管理员关联。
- 存储频道永远不会进入目标频道列表。
- 批量按钮错误返回具体行号和原因。
- 重复确认、已放弃和已过期统一返回安全提示，不泄露其他用户待确认内容是否存在。
- 频道自动绑定和状态暂停写入审计日志；存储频道排除事件不写审计。

## 测试策略

单元测试覆盖：

- 单帖和相册建立一个待确认记录。
- 默认命名、自定义命名、放弃、超时和重复确认。
- 批量按钮追加、同行列号、空行、非法格式、非法 URL、每行 8 个和总计 100 个边界。
- 槽位默认名称、自动同步、自定义名称保持、清空、移动和交换。
- 成员状态从非管理员到管理员、重复管理员事件、降权和移除。

SQLite 集成测试覆盖迁移、既有槽位名称回填、确认事务回滚、草稿配额、跨用户隔离、重启后待确认恢复和过期清理重试。

aiogram 适配测试覆盖 `ChatMemberUpdated` 字段映射、处理器注册带来的 `my_chat_member` allowed update、存储频道排除、权限检查、无法私聊时绑定仍成功，以及预览和确认回调数据长度。

完整测试套件必须继续验证现有发布顺序、防抖、自身消息过滤、批次恢复、频道共享和不可变版本行为。

## 验收标准

- 转发内容在用户确认前不出现在正式草稿列表，确认后只出现一次。
- 放弃或过期内容最终从存储频道删除；机器人重启不破坏确认和清理流程。
- 一次输入可原子添加多行、多列 URL 按钮。
- 1 至 10 号槽位始终显示可辨识名称，且个人草稿变化不影响已自定义名称。
- 机器人被添加为具备所需权限的频道管理员后，无需手填 ID 即可在邀请人的频道列表出现。
- 存储频道不会误绑定，降权和移除不会删除频道配置。
- 所有既有测试和新增测试通过。
